# 内存优化方案

> 版本：v1.0.1 | 更新日期：2026-06-17

## 一、问题描述

软件启动后初始内存占用即达到 100+ MB，执行 OCR 识别操作后内存占用持续超过 300 MB 且未有效释放。

### 根因分析

| 问题 | 内存占用 | 权重 |
|------|---------|------|
| ONNX Runtime 线程数 = 物理核数 (16) for inter+intra | 每线程 ~10-20MB arena 分配，约 100-150MB | 最大 |
| 模型数据字节缓存 det/rec/cls_model_data | 3 个 ONNX 文件 ~25-30MB 重复保存 | 高 |
| 图像 clone() + resize() 多份拷贝 | 单张 4K 截图 ~32MB × 3 = 96MB 峰值 | 高 |
| Engine 永不释放（ONNX session 持续占用） | 3 个 session ~80-120MB 常驻 | 高 |
| ONNX GraphOptimizationLevel::Level3 | 图优化分配大量中间内存 | 中 |

## 二、优化策略

**核心原则**：识别过程性能最大化，识别完成后及时释放。

### 2.1 ONNX Runtime 参数

| 配置项 | 值 | 说明 |
|--------|-----|------|
| inter_threads | `num_cpus::get_physical()` | 全部物理核心，最大并行推理 |
| intra_threads | `num_cpus::get_physical()` | 全部物理核心，最大算子内并行 |
| GraphOptimizationLevel | Level 3 | 最深图优化，推理速度最快 |

> 识别过程中内存占用会较大（300MB+），这是换取速度的必要代价。ONNX session 的线程 arena 无法按需缩容，只能通过整体释放 engine 来回收。

### 2.2 模型数据缓存

模型 ONNX 文件在首次创建 engine 时读入内存并缓存，避免每次识别都从磁盘读取（SSD 读取 ~0.5s，HDD 更久）。识别过程中缓存常驻，engine 释放时一并回收。

缓存字段（在 `OcrEngine` 结构体中）：
- `det_model_data: Vec<u8>` — 文本检测模型 (~4-6MB)
- `cls_model_data: Option<Vec<u8>>` — 方向分类模型 (~1-2MB)
- `rec_model_data: Vec<u8>` — 文本识别模型 (~15-20MB)

### 2.3 图像处理零拷贝

- `recognize_from_image` 获取图片所有权（`img: DynamicImage`），避免 `&` 引用方式导致必须 clone
- `ImageRgb8` 直接 move 使用，无需 clone
- `ImageRgba8` 使用 `into_raw()` 取走内部 buffer，并行转换后立即 `drop(raw)`
- resize 直接使用 owned 图片，不再 clone 原图片
- 推理完成后立即 `drop(image_buffer)`

### 2.4 Engine 自动释放机制

前端在三个时机调用 `ocr_release` 让 Rust 端丢弃 engine：

| 触发点 | 命令 | 说明 |
|--------|------|------|
| 离开 OCR 页面 | `useEffect cleanup → invoke("ocr_release")` | 切换到设置/关于时释放 |
| 批量识别完成 | `startBatchOCRForPaths 末尾` | 所有图片识别完释放 |
| 手动清空结果 | `handleClear` | 清空列表时释放 |

### 2.5 切换模型时自动释放

调用 `ocr_set_active_model` 切换 OCR 模型版本时，`ocr_engine` 字段被置空，下次识别会用新模型重建 engine。

## 三、预期效果

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 启动空闲 | ~100MB | ~30-40MB（仅前端 + 框架） |
| OCR 识别中 | ~300MB+ | ~300-400MB（**有意为之，换速度**） |
| OCR 完成后 | ~300MB+（不释放） | ~30-40MB（自动释放） |

## 四、注意事项

1. **识别中内存不回收属于正常行为** — ONNX Runtime 线程池 arena 分配了就不会缩小，只能整体 drop engine
2. **批量识别速度已最大化** — 物理核数 × Level3 × 模型缓存 = 最快组合
3. **识别完成后请点击"清空"或离开页面** — engine 自动释放，内存回落
4. **下次识别时重新创建 engine** — 模型缓存已启用，从磁盘加载 ~0.5s（SSD）
